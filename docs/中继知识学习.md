# 中继知识学习

## 1. 当前项目对接谁

`RelayAgentGo` 部署后，主要对接的是网络管理服务 `OwnZeroTierController`，不是直接管理 `zerotier-one` Controller。

但 `RelayAgentGo` 所在的中继机器上，通常也需要安装并运行 `zerotier-one` 客户端，让这台机器加入 ZeroTier 网络并获得 ZeroTier IP。

`ZT_NETWORK_ID` 现在是可选项。若 agent 不填写，controller 会在注册时返回它所归属的网络 ID，agent 再保存到本地 state。

整体关系：

```text
OwnZeroTierController
  管控制面：注册 relay、下发路由/NAT 配置、查看状态

RelayAgentGo
  跑在中继机上：上报状态、接收配置、修改本机路由和 nftables

zerotier-one
  跑在中继机上：让这台机器接入 ZeroTier 虚拟网络，获得 ZeroTier IP

Linux 内核
  真正转发流量：根据 ip route + nftables 做转发/NAT
```

可以简单理解为：

- `OwnZeroTierController` 是大脑。
- `RelayAgentGo` 是执行器。
- `zerotier-one` 提供接入 ZeroTier 网络的虚拟网卡和隧道能力。
- Linux 内核负责真正转发流量。

## 2. 大致部署方式

假设有一台中继服务器 `relay-01`。

### 控制器服务器

控制器服务器上部署：

```text
OwnZeroTierController
zerotier-one controller
MySQL
```

`OwnZeroTierController` 负责管理 ZeroTier 网络、成员授权，以及后续新增的 relay 管理接口。

对外只建议暴露：

```text
https://controller.example.com
```

不要把 `zerotier-one` 的 `9993` Controller API 直接暴露到公网。

### 中继节点

每台 relay 中继服务器上部署：

```text
zerotier-one
RelayAgentGo
nftables
iproute2
```

先让中继机加入 ZeroTier 网络：

```bash
zerotier-cli join <networkId>
```

然后在 `OwnZeroTierController` 里授权这个节点，让它获得 ZeroTier IP。

示例：

```text
公网 IP: 1.2.3.4
ZeroTier IP: 10.147.17.20
```

再启动 `RelayAgentGo`，配置它连接管理服务：

```env
CONTROLLER_BASE_URL=https://controller.example.com
CONTROLLER_TOKEN=xxxx
RELAY_NAME=relay-01
ZT_NETWORK_ID=
```

## 3. RelayAgentGo 如何产生作用

### 3.1 注册自己

`RelayAgentGo` 启动后，先向 `OwnZeroTierController` 注册自己。

上报内容大致包括：

```text
我是 relay-01
我的公网 IP 是 1.2.3.4
我的 ZeroTier IP 是 10.147.17.20
我的版本是 0.1.0
```

控制器收到后，会记录这个 relay 节点。

### 3.2 周期性上报状态

`RelayAgentGo` 会每隔一段时间上报状态，比如每 10 到 30 秒上报一次。

上报内容包括：

- 公网 IP。
- ZeroTier IP。
- CPU、内存、load。
- 网络延迟。
- 当前配置版本。
- `nftables` 是否应用成功。
- `ip route` 是否应用成功。

这样 `OwnZeroTierController` 就能知道这个 relay 是否在线、负载如何、配置是否生效。

### 3.3 拉取控制器配置

当控制器侧配置发生变化时，`RelayAgentGo` 会拉取最新配置。

控制器下发的配置大致包括：

```text
开启 IPv4 转发
给 10.20.0.0/24 加一条路由
给 10.147.17.0/24 做 MASQUERADE NAT
```

### 3.4 修改本机网络配置

`RelayAgentGo` 收到配置后，会在中继机上执行等价的本机网络配置。

示例：

```bash
sysctl -w net.ipv4.ip_forward=1

ip route replace 10.20.0.0/24 via 10.147.17.1 dev ztxxxxxx metric 100

nft add table ip relay_agent
nft add chain ip relay_agent postrouting ...
nft add rule ip relay_agent postrouting ip saddr 10.147.17.0/24 oifname eth0 masquerade
```

实际实现时，命令会由 Go 程序封装，并做幂等处理，避免重复创建冲突规则。

### 3.5 Linux 内核实际转发

配置完成后，真正转发流量的不是 Go 程序，而是 Linux 内核。

示例流量路径：

```text
ZeroTier 内网客户端
  -> 访问某个网段
  -> 流量进入 relay-01 的 ZeroTier 网卡
  -> Linux 根据 ip route 转发
  -> nftables 做 NAT
  -> 从公网网卡 eth0 出去
```

因此 `RelayAgentGo` 的角色是控制和配置，不是用户态流量代理。

## 4. 一句话总结

`RelayAgentGo` 部署在中继节点上，业务上对接 `OwnZeroTierController`，网络上依赖本机 `zerotier-one` 接入 ZeroTier 网络。它本身不搬运流量，只负责把控制器下发的路由和 NAT 策略变成本机 Linux 网络配置，真正转发由 Linux 内核完成。
